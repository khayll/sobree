# Sobree Render Corpus

The corpus is the foundation of Sobree's rendering fidelity work. Every
docx that hits a real user is the source of truth for whether the
renderer works. Synthetic test fixtures can't catch the long tail of
OOXML quirks that real documents accumulate over years of edits by
multiple authors.

## Layout

```
tests/corpus/
├── generated/         — single-feature synthetic fixtures (from @sobree/fixtures-gen)
│   ├── paragraph/     — heading levels, alignment, line spacing
│   ├── list/          — numbered, bulleted, multi-level
│   ├── table/         — borders, merged cells, headers
│   ├── header-footer/ — page numbers, even/odd, first-page
│   ├── tabs/          — left, right, center, decimal, leaders
│   └── mixed/         — combinations exercised together
├── real-world/        — actual docx files from users / public sources
│   ├── form/          — application forms, surveys
│   ├── letterhead/    — branded letters
│   ├── contract/      — legal templates
│   ├── cv/            — résumés
│   ├── academic/      — papers with citations, footnotes
│   ├── marketing/     — brochures, flyers
│   ├── recipe/        — cookbook-style mixed-content
│   └── report/        — technical / financial reports
└── README.md          — this file
```

## Per-corpus-file layout

Each docx lives in its category folder with companion artifacts:

```
<category>/<slug>/
├── source.docx           — the document (committed)
├── README.md             — what features it exercises, who contributed it, source URL if public
├── libreoffice/          — reference render (regenerated by `pnpm corpus:reference`)
│   ├── page-1.png
│   ├── page-2.png
│   └── metrics.json      — per-line text positions from PDF extraction
├── sobree/               — Sobree render (regenerated by `pnpm corpus:render`)
│   ├── page-1.png
│   ├── page-2.png
│   └── snapshot.json     — DOM block snapshot
└── baseline/             — committed reference numbers, updated explicitly
    ├── drift.json        — committed drift score (mean abs line-height delta)
    ├── known-issues.md   — what we know doesn't match Word, why
    └── tier.json         — { tier: 1 | 2 | 3, features: [...] }
```

## Tier definitions

- **Tier 1** — text, paragraphs, runs, sections, lists, simple tables, inline images, hyperlinks, basic headers/footers. Target: < 5% line-height drift vs LibreOffice.
- **Tier 2** — anchored shapes, columns, fields, footnotes, comments, revisions, complex tables, drop caps. Target: structural fidelity (right content on right page) without exact pixel parity.
- **Tier 3** — equations, charts, SmartArt, ActiveX, OLE. Best-effort fallback render or explicit "not supported" placeholder.

## Workflow

### Adding a corpus file

```sh
cp my-real-doc.docx tests/corpus/real-world/form/my-doc/source.docx
# Create tests/corpus/real-world/form/my-doc/README.md with feature notes
pnpm corpus:reference my-doc   # render via LibreOffice → libreoffice/
pnpm corpus:render my-doc      # render via Sobree → sobree/
pnpm corpus:diff my-doc        # compute drift, write baseline if first time
git add tests/corpus/real-world/form/my-doc/
```

### Running the full corpus

```sh
pnpm corpus:all                # render every file in both engines, diff all
pnpm corpus:diff               # diff-only (assumes renders are current)
pnpm corpus:diff --pages       # per-page allocation summary
```

### Updating baselines after intentional renderer changes

```sh
pnpm corpus:diff --update      # writes current diff scores as new baselines
git diff tests/corpus          # review the visual / numeric changes
git add -p
```

### CI gate

`pnpm corpus:check` runs the corpus and exits non-zero if any
fixture's drift score regresses beyond tolerance vs its committed
baseline. PRs cannot merge until either:

- the regression is fixed, or
- the baseline is explicitly updated with a justifying commit message.

## Why both generated and real-world?

- **Generated fixtures** isolate one OOXML feature each — failures point
  at exactly the cause.
- **Real-world docs** combine features the way users actually compose
  them, surfacing interaction bugs synthetic fixtures miss.

You need both. Generated catches regressions in narrow code paths;
real-world catches regressions in the layout engine's overall
behaviour.

## Contributing real-world docs

When adding a real doc:
1. **Strip PII** — replace names / addresses / IDs with placeholders. Keep structure intact.
2. **Document the source** in README.md (URL if public, anonymized "from user X" otherwise).
3. **License-check** — only add docs you have rights to redistribute. Public templates, OSS examples, your own work.
4. **Smallify** — if a doc has 50 pages of similar content, trim to 3-5 representative pages.
